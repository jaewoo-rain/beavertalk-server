# 11장. 전체를 하나로 — 통화 하나가 시스템을 관통하는 여정

> 여기까지 왔다면 부품 하나하나(FastAPI 골격, Supabase 연결, 인증 위임, 도메인 4계층, 실시간 통화)는 이제 익숙할 거예요. 이 마지막 장은 그 부품들을 **하나의 이야기**로 엮습니다. "사용자가 캐릭터에게 전화를 걸면, 그 한 통의 통화가 어떤 부품들을 거쳐 학습 데이터가 되는가?"를 끝까지 따라가 봅니다.

---

> 📘 **이 장을 읽고 나면**
>
> - 실시간 통화 한 건이 **연결 → AI 대화 → 저장 → 분석 → 복습**을 어떻게 관통하는지 그릴 수 있습니다.
> - 앞에서 배운 조각들이 **정확히 어느 순간** 등장하는지 압니다.
> - 프로젝트를 처음 열 때 **어느 파일부터 볼지** 실전 팁을 얻습니다.
> - 이 프로젝트의 **한계·주의점·다음 스텝**을 이해합니다.

---

## 1. 통화 한 건의 일생

시나리오: 사용자가 "이순신" 캐릭터에게 전화를 걸어 5분간 한국어로 대화합니다.

각 단계 옆에 **몇 장에서 배운 내용인지** 표시했습니다.

```
[클라이언트] Supabase SDK로 로그인 → JWT 획득                          (3장)
     │
     ▼  WebSocket 연결:  GET /api/v1/calls/stream?token=<JWT>
┌──────────────────────────────────────────────────────────────┐
│ ① 인증 + 회원 확보  (ws_router)                                 │  (3장·8장)
│   verify_token → Supabase 검증 → find_or_create_by_auth        │
├──────────────────────────────────────────────────────────────┤
│ ② 통화 준비  (call_session.run_call)                           │  (8장)
│   - 첫 start 메시지에서 character_id·locale 읽기               │
│   - DB에서 캐릭터 페르소나·레벨·관심사 로드 → 시스템 지시문 조립 │
│   - Call 행 생성 (status='ongoing')                            │  (2장 커밋·6장 모델)
├──────────────────────────────────────────────────────────────┤
│ ③ 실시간 대화  (asyncio.TaskGroup, 4개 pump 동시)              │  (8장)
│   ①client→Gemini(음성16k)  ②Gemini→client(음성24k+전사)       │
│   ③통화 시계(5분 후 작별 유도)  ④60초마다 DB flush             │
│      └ Gemini Live: 페르소나대로 대화/교정 + 음성 합성          │  (9장 외부연동)
├──────────────────────────────────────────────────────────────┤
│ ④ 종료  (call_ended 전송, status='analyzing')                  │  (8장)
└──────────────────────────────────────────────────────────────┘
     │ (백그라운드 태스크로 넘어감 — 클라이언트는 끊어도 됨)
     ▼
┌──────────────────────────────────────────────────────────────┐
│ ⑤ 통화 후 분석  (normalcall_service.analyze_call)              │  (8장)
│   - CallRawData로 대화 전문 재구성                             │
│   - Gemini 구조화 출력 → {요약, 모드, 배운 표현들}            │
│   - Sentence 행 생성 (source_type: asked/corrected/drilled)   │  (6장 모델)
│   - 각 표현 TTS 합성 → Supabase Storage 업로드                 │  (9장 스토리지)
│   - status='done'                                             │
└──────────────────────────────────────────────────────────────┘
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│ ⑥ 복습  (review_service)                                       │  (8장)
│   사용자가 문장을 다시 녹음 → SpeechSuper 발음 채점            │  (9장)
│   → Sentence.evaluation 갱신 (글자별 점수/등급)               │
└──────────────────────────────────────────────────────────────┘
```

이 한 장의 그림이 **BeaverTalk의 심장**입니다. 각 상자에서 "아, 이건 몇 장에서 봤지" 하고 떠올릴 수 있다면 이해도 80%는 넘긴 거예요.

### 핵심 통찰 3가지

1. **실시간과 배치는 완전히 다른 두 세계다.** 회원·캐릭터·결제·알람은 평범한 요청-응답(REST) CRUD이지만, 통화는 **WebSocket + 4개 동시 pump + 백그라운드 분석**이라는 별도의 우주입니다. 같은 백엔드 안에 두 패러다임이 공존해요.

2. **DB는 진실의 원천, 외부 AI는 갈아끼울 수 있는 부품.** 통화 세그먼트는 60초마다 DB에 저장되고(크래시 복원), 회원·인증·결제는 Postgres가 진실입니다. 반면 Gemini·TTS·SpeechSuper는 실패해도 앱이 죽지 않게 **graceful degradation**으로 감쌌습니다(9장).

3. **인증도 저장도 "위임"이 설계 철학.** JWT는 Supabase가, 파일은 Supabase Storage가, 발음채점은 SpeechSuper가 담당합니다. 우리 코드는 이들을 **조율(orchestrate)**할 뿐, 직접 구현하지 않아요. 덕분에 코드가 작고, 각 부품을 독립적으로 교체할 수 있습니다.

## 2. 프로젝트를 처음 열 때 — 어느 파일부터?

이 순서로 열어보세요.

6. **[domains/learning/realtime/call_session.py](../../domains/learning/realtime/call_session.py)** — 준비됐다면 플래그십(통화 엔진).

## 3. 이 프로젝트의 주의점·한계

정직한 마무리로, 지금 상태의 주의점을 짚습니다.

| 항목 | 내용 |
|------|------|
| **문서 드리프트** | `README.md`(자체 JWT)·`openapi.json`(옛 auth 엔드포인트)이 코드보다 뒤처짐. 코드가 진실. |
| **깨진 잔재** | `scripts/smoke_common.py`가 삭제된 `core.security`를 import. `bcrypt`/`pyjwt`는 미사용 잔재. |
| **Supabase 강한 결합** | 인증·DB·스토리지가 전부 Supabase. Supabase Auth가 없으면 인증 자체가 안 됨(유일한 hard dependency). |
| **실시간 통화의 상태성** | 통화는 서버 메모리에 세그먼트를 누적(60초마다 flush). 단일 인스턴스 가정에 가깝고, 수평 확장 시 통화 상태 공유는 별도 과제. |
| **외부 AI 비용/지연** | Gemini Live·TTS·분석은 실제 과금·지연이 있음. stub은 개발용. |
| **명시적 커밋의 함정** | service에서 `db.commit()`을 빼먹으면 조용히 저장 안 됨(자동 롤백). @Transactional에 익숙하면 실수하기 쉬움. |

이들은 대부분 **의도된 트레이드오프**(작은 코드로 빠르게)이거나 **진행 중 상태**입니다.

## 4. 다음 스텝 — 이제 뭘 하면 되나요?

1. **로컬 기동**: `alembic upgrade head` → `uvicorn main:app --reload` → http://localhost:8000/docs 로 API 둘러보기. dev 콘솔 `http://localhost:8000/__calldemo`로 통화 데모도 볼 수 있어요.
2. **작은 요청 따라가기**: `GET /api/v1/members/me`를 호출하고, 3장의 인증 흐름이 코드에서 어떻게 도는지 브레이크포인트로 확인.
3. **한 계층 고쳐보기**: account 도메인에 필드 하나 추가 → 모델 수정 → `alembic revision --autogenerate` → `upgrade` → DTO/서비스 반영. 4장·5장의 전체 사이클을 몸으로 익히기.
4. **드리프트 청소(선택)**: README의 인증 서술 갱신, 깨진 `smoke_common.py` 수정, 미사용 `bcrypt`/`pyjwt` 정리 — 좋은 첫 기여 과제입니다.
5. **더 깊은 문서**: 저장소 `docs/`의 설계 plan 문서들(normalcall, review-flow, call-history 등)로 각 기능의 "왜"를 파세요.
